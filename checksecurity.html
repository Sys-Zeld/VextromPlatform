Check Security

Passo a passo para validar as falhas revisadas e confirmar as correcoes no projeto atual.

Ferramentas Externas

  • Google Chrome ou Firefox com DevTools.
  • Postman ou Insomnia para repetir requests autenticados.
  • curl para provas simples via terminal.
  • Burp Suite Community ou OWASP ZAP para interceptar cookies, headers e CSRF.

Preparacao

  1. Suba a aplicacao com npm start ou npm run dev.
  2. Confirme a URL base em .env, usando APP_BASE_URL. Nos exemplos abaixo vou chamar de {{BASE_URL}}.
  3. Tenha um usuario admin valido para login em /admin/login.
  4. Tenha ao menos um token de formulario existente e, se possivel, um anexo PDF ja enviado.
  5. Se quiser testar o fluxo completo, crie uma API key na tela /admin/api-keys.

Matriz de Teste

Caso Ferramenta Resultado esperado
Cookie CSRF endurecido Navegador / Burp HttpOnly, SameSite=Strict e Secure quando HTTPS
API de equipamento bloqueada para API key comum Postman / curl 403 ADMIN_SESSION_REQUIRED
Download de documento sem token correto Navegador / curl 404 ou 403
Bypass de rate limit via X-Forwarded-For Burp / curl Header manual nao deve liberar criacao ilimitada
Erro 500 sem vazamento interno Navegador / Postman Mensagem generica, sem stack, path interno ou segredo

Passos Detalhados

1. Validar cookie do CSRF

  1. Acesse {{BASE_URL}}/admin/login.
  2. Abra DevTools em Application > Cookies ou intercepte no Burp.
  3. Localize o cookie de CSRF gerado pelo csurf.
  4. Confirme os atributos:
    HttpOnly=true
    SameSite=Strict
    Path=/
    Secure=true se {{BASE_URL}} for HTTPS

Falha antiga: cookie padrão sem endurecimento explícito.

Correção esperada: atributos visíveis no navegador/interceptador.

2. Validar que API key não acessa /equipment/:id/specification

  1. Crie ou use uma API key com escopo spec:read ou spec:write.
  2. No Postman ou curl, chame:
curl -i "{{BASE_URL}}/equipment/1/specification" ^
  -H "Authorization: Bearer SUA_API_KEY"
  1. Repita para PUT:
curl -i -X PUT "{{BASE_URL}}/equipment/1/specification" ^
  -H "Authorization: Bearer SUA_API_KEY" ^
  -H "Content-Type: application/json" ^
  -d "{\"values\":{}}"
  1. O retorno esperado e 403 com ADMIN_SESSION_REQUIRED.

Falha antiga: qualquer API key com escopo acessava qualquer equipamento por ID.

Correção esperada: somente sessão admin autenticada consegue usar esses endpoints.

3. Validar que download de documento exige o token correto

  1. Entre em um formulário e pegue um link de anexo no HTML de revisão:
    /form/TOKEN_REAL/documents/ID/download
  2. Teste no navegador com o token correto. O PDF deve baixar.
  3. Agora altere apenas o token na URL para um token inexistente ou de outro registro.
  4. Teste também trocar apenas o ID do documento.
curl -i "{{BASE_URL}}/form/TOKEN_ERRADO/documents/123/download"

Resultado esperado: 404 ou 403, nunca download do arquivo errado.

Falha antiga a evitar: front consumir URL pública/externa sem depender do token do formulário.

4. Validar que o front não usa mais externalUrl

  1. Abra /form/<token>/specification e /form/<token>/review.
  2. Inspecione os links dos anexos.
  3. Confirme que os anchors apontam para:
    /form/<token>/documents/<id>/download
  4. Confirme que não existe link clicável apontando para /dados/docs/....

Ferramenta: navegador, DevTools ou View Source.

5. Validar resistência a spoof de X-Forwarded-For

  1. Use Burp ou curl para chamar várias vezes {{BASE_URL}}/form/public-start.
  2. Altere manualmente o header X-Forwarded-For a cada request.
  3. Observe se o limite continua contando pelo IP resolvido da aplicação e pela sessão de navegador.
curl -i "{{BASE_URL}}/form/public-start" -H "X-Forwarded-For: 1.1.1.1"
curl -i "{{BASE_URL}}/form/public-start" -H "X-Forwarded-For: 2.2.2.2"
curl -i "{{BASE_URL}}/form/public-start" -H "X-Forwarded-For: 3.3.3.3"

Falha antiga: o sistema aceitava esse header bruto do cliente.

Correção esperada: mudar o header manualmente não deve ser suficiente para burlar o controle.

Observação: como também existe hash de sessão de navegador, valide em janela anônima e janela normal para comparar.

6. Validar sanitização de erro 500 no envio de e-mail

  1. Configure o SMTP de forma inválida de propósito em ambiente de teste, ou interrompa o serviço SMTP.
  2. Acesse /form/<token>/review.
  3. Envie o formulário de e-mail em /form/<token>/send-email.
  4. Observe a resposta HTML.

Resultado esperado: mensagem genérica de falha, sem stack trace, sem hostname SMTP, sem path interno e sem segredo.

Falha antiga: retorno direto de err.message.

7. Validar handler final de erro sem vazamento

  1. Provoque um erro 500 em ambiente de teste. O jeito mais seguro é usar dependência externa inválida, como SMTP quebrado, ou desconectar banco em uma rota não crítica de teste.
  2. Faça a chamada pelo navegador e pelo Postman.
  3. Confirme que a resposta ao cliente contém apenas mensagem genérica.
  4. Confirme que detalhes técnicos aparecem apenas no log do servidor.

Coletar evidência: screenshot da resposta HTTP + trecho do log do servidor.

8. Validar fluxo admin de manutenção e backups

  1. Acesse {{BASE_URL}}/admin/maintenance sem login.
  2. O esperado é redirecionamento para /admin/login.
  3. Entre com usuário comum sem role admin.
  4. O esperado é 403 ao tentar manutenção.
  5. Entre com usuário admin e confirme acesso liberado.

Objetivo: validar separação entre autenticação e autorização por papel.

9. Validar upload de PDF com sessão expirada ou token bloqueado

  1. Use um token cujo status esteja sent.
  2. Tente abrir a especificação sem sessão admin.
  3. O esperado é redirecionamento para revisão com aviso de bloqueio.
  4. Tente subir um PDF para esse mesmo token.

Resultado esperado: 403 SPECIFICATION_LOCKED no upload.

Checklist de Evidências

  • Screenshot dos cookies de sessão/CSRF.
  • Resposta do Postman/curl para /equipment/:id/specification com API key.
  • Teste de download com token correto e token incorreto.
  • Teste repetido de /form/public-start alterando X-Forwarded-For.
  • Resposta de erro 500 sanitizada no envio de e-mail.
  • Logs do servidor mostrando detalhe interno apenas no backend.

Critério de Aprovação

Teste Aprovado quando
CSRF cookie Tem HttpOnly, SameSite=Strict e Secure em HTTPS.
API por ID API key sozinha recebe 403 ADMIN_SESSION_REQUIRED.
Documentos Somente link controlado por token funciona; token/ID errados não baixam arquivo.
Rate limit público Mudar X-Forwarded-For manualmente não destrava criação ilimitada.
Erros 500 Sem stack trace, sem SQL, sem caminhos internos, sem segredos na resposta.
