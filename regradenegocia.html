Regra de Negocio do Sistema

Aplicacao: APP-FORM-UPS-IEC

Baseado no comportamento implementado em src/app.js e nos servicos de dominio do projeto.

Formulario dinamico Perfis reutilizaveis Tokens por cliente Fluxo publico controlado PDF, e-mail e anexos

1. Objetivo do sistema

O sistema gerencia a coleta de especificacoes tecnicas de UPS com base no Anexo D da IEC 62040-3. Ele permite montar formularios dinamicos, associar conjuntos de campos a perfis, gerar tokens individuais por cliente e consolidar a resposta final em tela de revisao, PDF, JSON e envio por e-mail.

2. Entidades principais

Entidade Papel de negocio
fields Catalogo mestre de campos do formulario. Cada campo tem secao, tipo, unidade, ordem e opcionalmente valor padrao.
field_profiles e field_profile_fields Perfis reutilizaveis que definem quais campos ficam habilitados e como cada campo pode ser customizado dentro do perfil.
equipments Registro de cada solicitacao de cliente. Guarda token, dados do cliente, perfil aplicado e status do fluxo.
equipment_enabled_fields Lista efetiva de campos habilitados para um token especifico.
equipment_field_values Valores realmente preenchidos e salvos para cada campo de um equipamento.
equipment_documents Anexos PDF vinculados ao token.
public_token_links Links publicos que permitem iniciar o preenchimento usando um perfil predefinido.
api_keys, admin_users e sessoes Camada de autenticacao e autorizacao do painel e das APIs.

3. Logica central de preenchimento

  1. O admin define os campos base do sistema em /admin/fields.
  2. O admin cria perfis em /admin/profiles, podendo habilitar ou desabilitar campos, alterar label, secao, tipo, obrigatoriedade, ordem e default dentro do perfil.
  3. Ao cadastrar um cliente em /admin/clients/new, o sistema cria um equipment com token unico e associa os campos habilitados.
  4. O formulario de um token mostra apenas os campos efetivamente habilitados para aquele equipamento.
  5. O preenchimento pode ser salvo parcialmente enquanto o status estiver em draft.
  6. Na etapa de revisao, o usuario pode gerar PDF e enviar e-mail. Quando o e-mail e enviado com sucesso, o token muda para sent.
  7. Depois de sent, o formulario fica bloqueado para o publico e so pode ser alterado por sessao admin autenticada.

4. Regra de valor efetivo dos campos

A exibicao de cada campo no formulario segue uma precedencia fixa implementada em src/services/specifications.js:

  1. Se existir valor salvo em equipment_field_values, esse valor prevalece.
  2. Se nao existir valor salvo e o campo tiver hasDefault = true, o sistema usa defaultValue.
  3. Se nao houver valor salvo nem default, o campo aparece vazio.

Ao salvar vazio, o registro persistido do campo e removido e a resolucao volta a depender de default ou vazio.

5. Tipos de campo e validacoes

Tipos suportados

  • text: texto livre.
  • number: converte numero, inclusive com virgula.
  • enum: aceita apenas opcoes cadastradas.
  • boolean: aceita verdadeiro/falso em formatos padronizados.
  • time: exige formato hh:mm.
  • dimension: exige formato equivalente a HxWxD.

Regras associadas

  • Campos enum precisam de lista de opcoes.
  • Se hasDefault for verdadeiro, o default precisa ser valido para o tipo do campo.
  • A chave key do campo precisa ser unica e seguir padrao slug em minusculas com underscore.
  • Os erros de validacao retornam 422 nas APIs ou reexibem a tela com mensagens no fluxo web.

6. Fluxos de negocio

6.1 Fluxo administrativo principal

  1. Admin autentica em /admin/login.
  2. Admin cria ou ajusta campos base.
  3. Admin cria ou ajusta perfis de formulario.
  4. Admin cadastra um cliente, opcionalmente aplica um perfil e pode salvar a selecao atual como novo perfil.
  5. O sistema gera um token unico para o cliente acessar o formulario.

6.2 Fluxo do cliente por token

  1. Cliente acessa /form/:token/specification.
  2. Preenche dados do cliente e campos tecnicos habilitados.
  3. Pode salvar e continuar ou seguir para revisao.
  4. Na revisao, pode baixar PDF ou enviar o formulario por e-mail.
  5. Quando o e-mail e enviado com sucesso, o status vira sent.

6.3 Fluxo publico

  1. Um link publico ativo aponta para um perfil predefinido.
  2. Ao iniciar, o sistema valida limite por IP e sessao de navegador.
  3. Se liberado, gera um token temporario de rascunho.
  4. Ao enviar o formulario, o sistema cria um equipment real e salva a especificacao.

7. Regras sobre perfis

  • Perfil e um molde reutilizavel do formulario.
  • Um perfil nao define apenas selecao de campos; ele tambem pode sobrescrever configuracoes do campo no contexto do perfil.
  • Ao aplicar um perfil a um cliente, os IDs dos campos habilitados sao copiados para o equipamento.
  • Depois disso, a configuracao do cliente pode ser personalizada sem alterar o perfil original.
  • O sistema tambem permite criar campos novos diretamente dentro do perfil, gerando uma key unica automaticamente.

8. Regras sobre status do token

Status Significado Efeito pratico
draft Formulario ainda esta em elaboracao. Pode continuar recebendo edicoes pelo link do token.
sent Submissao considerada enviada/finalizada. Edicao publica e bloqueada; apenas admin autenticado pode alterar.

9. Documentos anexos

  • Somente PDF e aceito.
  • Limite por arquivo: 10 MB.
  • Limite por equipamento: 10 documentos.
  • O sistema valida assinatura basica do arquivo PDF antes de salvar.
  • O arquivo vai para dados/docs e o metadado fica salvo no banco.
  • O download e feito por rota controlada, nao por exposicao irrestrita de pasta.

10. PDF, revisao e envio por e-mail

  • A revisao consolida os dados do cliente, os campos respondidos e os documentos anexos.
  • O PDF e gerado sob demanda a partir da especificacao atual do token.
  • O envio de e-mail usa SMTP configurado no ambiente e anexa o PDF da submissao.
  • Existe rate limit para envio de e-mail, reduzindo abuso da funcionalidade.

11. IA para criacao de perfis

  • O modulo de IA recebe documento PDF, TXT ou planilha Excel e um modelo JSON de perfil.
  • A OpenAI e usada para extrair um JSON no formato exigido pelo sistema.
  • O retorno precisa conter name e fields; caso contrario, a importacao e rejeitada.
  • Se a resposta vier truncada por limite de tokens, o sistema tenta novamente com mais capacidade de saida.

12. Autenticacao, autorizacao e seguranca

  • Painel admin usa sessao assinada por cookie.
  • Rotas web protegidas usam CSRF.
  • APIs REST aceitam API key com escopos como fields:read, fields:write, spec:read e spec:write.
  • Sessao admin valida tambem pode operar as APIs internas como fallback.
  • O fluxo publico tem controle de taxa por IP e sessao de navegador, com janela de 12 horas.
  • Cabecalhos de seguranca sao reforcados com helmet.

13. Regras operacionais e manutencao

  • O sistema executa migracao do banco na subida da aplicacao.
  • O seed do Anexo D e aplicado no startup para garantir estrutura minima.
  • O painel de manutencao permite backup, restore, gestao de API keys, usuarios admin e operacoes auxiliares de tokens.
  • Existe catalogo de backups e suporte a importacao de arquivo .sql pela interface administrativa.

14. Resumo executivo da regra de negocio

O sistema foi desenhado para transformar um catalogo dinamico de campos em formularios controlados por perfil e por cliente.

A regra principal e: perfil define a estrutura, token identifica a solicitacao, especificacao guarda os valores, e o status controla se o formulario ainda pode ser editado.

Os comportamentos de bloqueio, default, validacao tipada e restricao de documentos sao partes centrais da logica e nao devem ser alterados sem avaliar impacto no fluxo completo.
